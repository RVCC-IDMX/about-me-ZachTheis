CSS Image Hover Effects

Much like links, images in CSS can be manipulated by users running a website using the :hover pseudoclass. Hover effects can be as simple as hiding or revealing an image or can involve more complex transformations or even animations. The Photo Gallery, above, shows six examples of this, though many, many more are possible. While I would typically use the same image hover effect for all images in the gallery, in the interest of demonstrating several techniques, each of the above images demonstrates a different effect.

Most image hover effects are implemented by giving the image a div tag with the class “container,” though this is not necessary. For my gallery, since all of the images were contained as figures, I gave each figure a separate class – from img-1 through img-5 – in place of a container class. The container is typically the element that triggers the hover state in its child elements.

The first image in the gallery creates an overlay with a fade-in effect. The overlay itself is a separate div which can be styled in any number of ways. For simplicity, my overlay is a solid box that matches the image in size and a border radius when the containing figure has one. The effect is achieved by setting the overlay’s opacity to 0, making it fully transparent, then changing the opacity to 1 when the mouse hovers over the container.

Images 2 and 3 change the image using the rendering engine of the browser. Each browser has its own rendering engine: Webkit for Safari, Blink for Chrome and Opera (older versions of Opera use Presto), Gecko for Firefox, Trident for Internet Explorer, and EdgeHTML for Edge. Each rendering engine operates differently and requires a different command, so best practices are to include the rendering commands for multiple browsers. This can be seen in styles.css; all commands to the rendering engine are repeated several times with different prefixes to instruct different browsers, which are referred to as vendor prefixes (-webkit-, -ms-, -moz-, and -o-). It’s good practice to also include the same property with no prefix. Doing this allows the browser you’re running in to use its native rendering engine if it supports the property.

In Image 2, the rendering engine applies a 3-pixel blur to the image upon hover. Image 3 applies a sepia filter, rendering the image in shades of tan. In both cases, a .3s ease-in-out value for the transition property is used, causing the image to transition to and from the hover state in 0.3 seconds. The same transition effect is achieved with a different implementation in later images.

Image 4 creates an animation hover effect, a transparent white circle that starts in the center of the image, rapidly expands, then vanishes. This uses the rendering engine’s animation property, which requires keyframes to be defined. These allow the programmer to set the animation’s state at intermediate points of the animation, defined by percentages of the animation’s completion, allowing for much more control over the effect than the transition property. As with all properties that rely on the rendering engine, keyframes require the use of a prefix to instruct a specific rendering engine and should be followed with a set of keyframes with no prefix.

Images 5 and 6 alter the display of their images, adding a tilt and a border-radius respectively. Since these properties are supported in all browsers, they do not require vendor prefixes. In these examples, an equivalent effect to ease-in-out has been achieved with the transition property on both the native and hover states of the container, using the same value for each.
